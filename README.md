# clusterssh — Bloom Leukemia Classifier

Leukemia subtype classifier built on top of **DinoBloom-G** — the original pretrained ViT-Giant model trained on 13M+ hematology images. We add a classification head and fine-tune the top layers on our dataset. The original DinoBloom-G is never modified; our output is a separate model (`bloom_leukemia.pth`).

**Windows only** for local use (`.bat` files). Cluster runs Linux (`.sh` files).

- Cluster: `rpatel1@login-01.ncshare.org`
- Storage: Oracle Object Storage (`bloomi-training-data`, region `us-ashburn-1`)
- Dataset: Google Drive folder (public) — `https://drive.google.com/drive/folders/1J5ld-tK6cewj9wXWUi3rs6UdlHnDBe8U`
- DinoBloom-G (original base model): Google Drive — `https://drive.google.com/file/d/1lLbicaSSUHDy0X9_o-XmareFf0rj2Bma/view`

## Dataset Format

Images follow the **C-NMC naming convention**:

```
{patient_id}_{slide_id}_{image_id}_{CLASS}.{ext}

Examples:
  4_7_400_ALL.bmp    →  class: ALL  (Acute Lymphoblastic Leukemia)
  2_1_150_HEM.bmp    →  class: HEM  (Healthy / Hematogone)
  9_3_220_Early.bmp  →  class: Early
```

The class label is the **last underscore-separated segment** of the filename (before the extension). All scripts — `make_splits.py`, `train_efficientnet_b0.py`, and `train_efficientnet_b0_ddp.py` — parse the class from the filename using this rule, with the parent folder name as fallback.

Archives are stored in `~/bloomi/New Data/extracted/` on the cluster:

```
New Data/
  extracted/
    archive5/   ← pre-split via train.txt / val.txt (if present in Drive)
    archive6/   ← auto-discovered, 80/20 split generated by make_splits.py
    archive7/
    archive8/
    archiveN/   ← any new archives from Google Drive are picked up automatically
  train.txt     ← generated by make_splits.py if missing
  val.txt
```

## Workflow (Windows)

### Step 1 — First-time setup
```cmd
.\setup.bat
```

The **only command needed on a fresh machine or after new archives are added**. It:

1. **Pre-flight** — tests SSH connectivity, shows cluster disk usage, shows existing archives
2. **Git pull** — ensures local scripts are up to date
3. **SSH key cache** — enter passphrase once, reused for all steps
4. **Launcher scripts** → `~/` (setup.sh, train_h200.sh, epoch_report.py)
5. **Training scripts** → `~/bloomi/` (train scripts, make_splits.py, requirements.txt)
6. **dinov2/** → `~/bloomi/` (skips if already present)
7. **Google Drive sync** — runs `gdown --remaining-ok` on the cluster to download any missing archives from the public Drive folder; shows before/after inventory with sizes
8. **Image count + split generation** — runs `make_splits.py` on cluster:
   - Prints total image count per archive and per class
   - Skips if `train.txt` / `val.txt` already exist
   - Auto-generates stratified 80/20 split if missing
9. **OCI key** → `~/.oci/oci_api_key.pem` (skips if already present)
10. **Clears DinoBloom-G weights** — fresh download by setup.sh
11. **Runs setup.sh** on cluster — installs OCI CLI, downloads weights, builds venv

### Step 2 — Submit training job
```cmd
.\train_h200.sh                REM 1x H200 (default)
sbatch train_h200.sh 2         REM 2x H200
sbatch train_h200.sh 4         REM 4x H200 (requires unc_h200 QOS from sysadmin)
```

### Step 3 — Monitor
```cmd
ssh rpatel1@login-01.ncshare.org "squeue -u rpatel1"
ssh rpatel1@login-01.ncshare.org "tail -f ~/bloomi/logs/dino_<JOBID>.out"
```

## Training Configuration (1x H200)

| Setting | Value | Why |
|---------|-------|-----|
| Epochs | **75** | Sweet spot for ViT-G fine-tuning — converges without overfitting |
| Batch size | **64** | Fills ~50GB of H200's 96GB VRAM with AMP |
| Learning rate | **1e-4** (head), **1e-5** (backbone) | Differential LR — backbone adapts gently |
| Unfreeze blocks | **4 of 40** | Preserves hematology pretraining, adapts top layers |
| Workers | **16** | Saturates data pipeline on 56-CPU node |
| Scheduler | CosineAnnealingLR | Smooth LR decay over 75 epochs |
| Label smoothing | 0.1 | Prevents overconfidence on small classes |
| Mixed precision | AMP (float16) | ~2× throughput, halves VRAM |

> **Extend to 100 epochs** only if val accuracy is still rising at epoch 75 — also update `T_max=100` in the training script's `CosineAnnealingLR`.

## Smart Split Generation (`make_splits.py`)

After every Google Drive sync, `setup.bat` runs `make_splits.py` on the cluster, which:

- Scans **all** `archiveN` folders automatically (no hardcoded list)
- Parses class from each filename (`4_7_400_ALL.bmp` → `ALL`)
- Prints a full inventory table: images per archive, images per class, total
- **If `train.txt` / `val.txt` exist** → reports sizes and exits (no overwrite)
- **If missing** → generates a **stratified 80/20 split** so each class is proportionally represented in both train and val

Example output:
```
  Archive          Images
  ───────────────  ──────────
  archive5         12,500
  archive6          8,200
  archive7          9,800
  ───────────────  ──────────
  TOTAL            30,500

  Class                           Images
  ──────────────────────────────  ──────────
  ALL                             18,300
  HEM                             12,200

  Generating stratified 80/20 split...
  train.txt : 24,400 images
  val.txt   :  6,100 images
```

## Oracle ↔ H200 Flow

```
Google Drive (public) — single file
    │
    │  setup.bat step [4] — cluster downloads via gdown, extracts,
    │                        then uploads to Oracle for future use
    ▼
Oracle bucket: bloomi-training-data/extracted/main_dataset.*
    │
    │  setup.sh step [3] — pulls from Oracle (fast, same cloud),
    │                       falls back to Google Drive if not yet in Oracle
    ▼
~/bloomi/New Data/extracted/   (auto-extracted, any format)

Google Drive (public file)
    │
    │  setup.sh step [4] — gdown downloads DinoBloom-G.pth directly
    ▼
~/bloomi/DinoBloom-G.pth  (~4.4 GB pretrained weights)

Oracle bucket: bloomi-training-data
    │
    │  [H200 trains for 75 epochs]
    │
    │  train_h200.sh pushes UP every epoch:
    ├─ trained-models/unc-h200/job<ID>_<date>/backups/checkpoint_epoch_NNN.pth
    │
    │  train_h200.sh pushes UP on completion:
    ├─ trained-models/unc-h200/job<ID>_<date>/best.pth    (bloom_leukemia.pth at best val acc)
    ├─ trained-models/unc-h200/job<ID>_<date>/last.pth    (final epoch checkpoint)
    └─ trained-models/unc-h200/job<ID>_<date>/run_info.txt
```

No models accumulate on cluster disk — all checkpoints are uploaded to Oracle and deleted locally.

**Download trained model:**
[cloud.oracle.com](https://cloud.oracle.com) → Storage → Object Storage → `bloomi-training-data` → `trained-models/unc-h200/` → your job folder → `best.pth`

## Epoch Reports

Every **5 epochs**, a full diagnostic block prints to the SLURM log (`logs/dino_<jobid>.out`):

- Timing: epoch time, elapsed, ETA, progress bar
- Loss & accuracy: train/val, best so far, per-class breakdown
- Loss trend table across all recorded epochs
- Optimizer state: LR per param group, weight decay, betas
- Weight stats: per layer — mean, std, L2 norm, % near-zero
- Gradient stats: per layer — mean, std, norm, % NaN, vanishing/exploding warnings
- GPU memory: allocated, reserved, peak, total VRAM
